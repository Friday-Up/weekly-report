# 工作周报生成 Skill

## 概述

从飞书汇报（学习日报）获取指定周的每日工作记录，按照固定模板和写作风格，生成一篇完整的工作周报并保存为 Markdown 文件。

## 前置：读取配置

在开始之前，先读取 skill 目录下的 `config.json` 获取用户信息：

```bash
cat <skill目录>/config.json
```

如果 `config.json` 不存在，提示用户参照 `config.example.json` 创建。

配置字段说明：
- `user_open_id`：飞书用户 Open ID（用于 API 查询）
- `author_name`：作者姓名
- `author_title`：职位/角色
- `company`：公司名称
- `mobile`、`email`、`address`：签名信息
- `file_prefix`：周报文件名前缀

## 流程

### 第 1 步：确定日期范围

根据用户输入确定周报的日期范围（周一到周五）：

- 用户说"本周周报"→ 取本周一到周五的日期
- 用户说"上周周报"→ 取上周一到周五的日期
- 用户给出具体日期如"4.20-4.24"→ 直接使用
- **用户给出具体年份如"2026年的4.27-2026年的4.30"→ 使用用户明确指定的年份，不可默认使用当前年份**

**重要：计算 Unix 时间戳前，必须先确认用户是否指定了年份。**
- 如果用户输入包含年份（如"2026年4.27"、"2026/4/27"），必须使用用户指定的年份。
- 如果用户未指定年份（如"4.27-4.30"），默认使用当前年份。
- **严禁假设当前年份**：在计算时间戳前，先用 `date` 命令验证当前年份，并与用户输入对比。若用户明确写了年份但与你默认使用的年份不同，必须以用户输入为准重新计算。

计算对应日期的 Unix 时间戳（北京时间 +08:00），起始日 00:00:00 到结束日 23:59:59。建议用以下命令验证时间戳对应的日期是否正确：

```bash
# 验证时间戳是否正确对应目标日期
date -r <时间戳> "+%Y-%m-%d %H:%M:%S"
```

### 第 2 步：获取飞书日报数据

通过 lark-cli 调用飞书汇报 API 获取该周的日报（`user_id` 从 config.json 读取）：

```bash
lark-cli api POST /open-apis/report/v1/tasks/query \
  --params '{"user_id_type":"open_id"}' \
  --data '{"commit_start_time":"<起始时间戳>","commit_end_time":"<结束时间戳>","user_id":"<config.user_open_id>","page_size":20,"page_token":""}'
```

如果返回 `has_more: true`，继续用 `page_token` 翻页直到获取完整数据。

从返回数据中提取每天的：
- `今日总结`（field_name 为 "今日总结"）
- `待办`（field_name 为 "待办"）
- `备注` / `明日计划`（field_name 可能是 "备注" 或 "明日计划"）

注意：日报内容是纯文本，状态标记用 emoji：✅已完成、🔵进行中、❌未开始、⏸已暂停。

### 第 3 步：分析和归类内容

将一周内所有日报的内容进行分析和归类：

1. **识别核心工作线**：从每日总结中提取本周的主要工作方向（通常 2-3 条主线）
2. **合并同类工作**：同一项目不同天的进展合并到一起，按进展脉络梳理
3. **区分完成/进行中/待推进**：
   - ✅ 标记的是已完成项，作为"成果"素材
   - 🔵 标记的是进行中，作为"进展"素材
   - ❌ 标记的是待推进，作为"待推进事项"素材
4. **内容过滤与去噪**：
   - 过滤掉无实质产出的日常活动（参会、阅读、学习、调研等）
   - 过滤掉碎片化琐事（单字段修复、样式调整、配置更新等），除非属于系统性修复或紧急需求
   - 过滤掉无具体成果的跟进事项（"持续推进"、"跟进进度"等空泛表述）
   - 同一项目多天的进展必须合并，禁止按天罗列
5. **信息密度控制**：
   - 每个工作线的进展子分类不超过 3 个
   - 单条 bullet 描述不超过 30 字，禁止展开技术细节
   - 一句话能说完的琐事不单独成条，合并到相关大项作为附注或直接省略
6. **提取下周计划线索**：从待办字段、❌未完成项、🔵进行中项推断下周重点

### 第 4 步：按模板生成周报

严格按照以下模板结构生成周报内容（签名部分从 config.json 读取）：

```
各位领导、管培生小伙伴们：

大家好！本周（M.DD-M.DD）主要围绕{核心工作线1}、{核心工作线2}等相关工作展开，{一句话概括本周重点}，现将具体情况总结如下：
 
一、核心任务
1. {工作线1名称}
    - 进展：
      - {子分类1}
        - {具体进展描述}；
        - {具体进展描述}；
      - {子分类2}
        - {具体进展描述}；
      - 待推进事项
        - {未完成的工作}；
    - 成果：
      - {成果1，概括性描述，突出价值}；
      - {成果2}；
    
2. {工作线2名称}
    - 进展：
      - {子分类1}
        - {具体进展描述}；
      - 待推进事项
        - {未完成的工作}；
    - 成果：
      - {成果描述}；
 
二、学习与思考
 
1. {主题}：{结合本周工作提炼的思考，150字左右}。
2. {主题}：{思考}。
3. {主题}：{思考}。 
 
三、下周工作计划
1. 【高优先级】{计划描述}。
2. 【高优先级】{计划描述}。
3. 【中优先级】{计划描述}。
4. 【低优先级】{计划描述}。


————————————

{config.author_name}

{config.company}

Mobile：{config.mobile}

E-mail：{config.email}

Ad：{config.address}

[图片]
```

### 写作风格要求

参照真实周报的语言风格，保持以下特征：

**开头段**：
- 固定格式"各位领导、管培生小伙伴们"
- 概括本周核心工作线（2-3个），加一句聚焦描述
- 例："聚焦任务落地与问题攻坚"、"聚焦核心任务落地，同步推进各项待办事项"

**核心任务-进展**：
- 按子分类组织，不要罗列日报原文
- 把同一项目不同天的碎片化进展合成连贯的段落描述
- 精炼概括：日报中的冗长描述要压缩提炼，去掉重复信息和过多细节，一句话能说清的不拆两句
- 用"完成XXX"、"推进XXX"、"明确XXX"等动词开头
- 技术细节保留但要概括，不要逐条复制日报
- "待推进事项"放在进展最后，用"未完成"、"暂未完成"等措辞

**核心任务-成果**：
- 高度概括，突出价值和意义
- 每条成果 1-2 句话
- 常用句式："{做了什么}，为{后续目标}奠定基础/提供支撑"

**学习与思考**：
- 3 条，每条一个小标题加冒号
- 结合本周具体工作提炼方法论
- 小标题用四字/六字总结如"追根溯源、精准突破"
- 每条 120-180 字，包含具体事例 + 总结感悟
- 语气正式但不空洞，要有真实工作的痕迹

**下周工作计划**：
- 从本周未完成项（❌）、进行中项（🔵）、待办字段智能推断
- 按优先级标注【高优先级】【中优先级】【低优先级】
- 高优先级：本周核心任务的延续/未完成项
- 中优先级：持续进行但非紧急的工作
- 低优先级：可延后或辅助性工作
- 每条用分号连接多个子任务

### 内容过滤与质量控制规则

**必须过滤掉的内容（不得写入周报）**：
- 日常琐事：参加会议、阅读文章/技术动态、准备分享会素材、编写培训文档等无直接业务产出的活动
- 碎片化小事：修复列表排序、调整 UI 样式、更新配置、排查已解决的临时问题等，除非属于紧急需求或系统性修复
- 无实质进展的跟进事项："持续推进 XXX"、"跟进 XXX 进度"等无具体成果的表述
- 个人学习行为：学习新技术、调研工具、体验产品等，除非已形成可落地的业务方案

**信息密度控制**：
- 每个工作线的进展子分类不超过 3 个
- 单条 bullet 描述不超过 30 字，禁止展开技术细节
- 同一项目多天的进展必须合并为一条连贯描述，禁止按天罗列
- 一句话能说完的琐事不单独成条，可合并到相关大项作为附注或直接省略

**价值判断标准**：
- 只保留有实质性进展、业务价值或关键决策的事项
- 成果部分必须体现对业务的实际影响（如"保障核心流程不中断"、"提升操作效率"），不能只是"完成了 XXX"
- 待推进事项只保留对下周有指导意义的未完成任务，已关闭或优先级过低的事项不写入

### 第 5 步：保存文件

将生成的周报保存到 skill 目录下的 `docs/` 子目录中，文件名格式：

```
{config.file_prefix}（M.DD-M.DD）.md
```

其中日期使用不带前导零的格式，如 `3.16-3.20`、`4.20-4.24`。

保存后告知用户文件路径，并询问是否需要调整。
