# 工作周报生成 Skill

## 概述

从飞书汇报（学习日报）获取指定周的每日工作记录，按照固定模板和写作风格，生成一篇完整的工作周报并保存为 Markdown 文件。

## 前置：读取配置

在开始之前，先读取 skill 目录下的 `config.json` 获取用户信息：

```bash
cat <skill目录>/config.json
```

如果 `config.json` 不存在，提示用户参照 `config.example.json` 创建。

配置字段说明：
- `user_open_id`：飞书用户 Open ID（用于 API 查询）
- `author_name`：作者姓名
- `author_title`：职位/角色
- `company`：公司名称
- `mobile`、`email`、`address`：签名信息
- `file_prefix`：周报文件名前缀

## 流程

### 第 1 步：确定日期范围

根据用户输入确定周报的日期范围（周一到周五）：

- 用户说"本周周报"→ 取本周一到周五的日期
- 用户说"上周周报"→ 取上周一到周五的日期
- 用户给出具体日期如"4.20-4.24"→ 直接使用

计算对应日期的 Unix 时间戳（北京时间 +08:00），起始日 00:00:00 到结束日 23:59:59。

### 第 2 步：获取飞书日报数据

通过 lark-cli 调用飞书汇报 API 获取该周的日报（`user_id` 从 config.json 读取）：

```bash
lark-cli api POST /open-apis/report/v1/tasks/query \
  --params '{"user_id_type":"open_id"}' \
  --data '{"commit_start_time":"<起始时间戳>","commit_end_time":"<结束时间戳>","user_id":"<config.user_open_id>","page_size":20,"page_token":""}'
```

如果返回 `has_more: true`，继续用 `page_token` 翻页直到获取完整数据。

从返回数据中提取每天的：
- `今日总结`（field_name 为 "今日总结"）
- `待办`（field_name 为 "待办"）
- `备注` / `明日计划`（field_name 可能是 "备注" 或 "明日计划"）

注意：日报内容是纯文本，状态标记用 emoji：✅已完成、🔵进行中、❌未开始、⏸已暂停。

### 第 3 步：分析和归类内容

将一周内所有日报的内容进行分析和归类：

1. **识别核心工作线**：从每日总结中提取本周的主要工作方向（通常 2-3 条主线）
2. **合并同类工作**：同一项目不同天的进展合并到一起，按进展脉络梳理
3. **区分完成/进行中/待推进**：
   - ✅ 标记的是已完成项，作为"成果"素材
   - 🔵 标记的是进行中，作为"进展"素材
   - ❌ 标记的是待推进，作为"待推进事项"素材
4. **提取下周计划线索**：从待办字段、❌未完成项、🔵进行中项推断下周重点

### 第 4 步：按模板生成周报

严格按照以下模板结构生成周报内容（签名部分从 config.json 读取）：

```
各位领导、管培生小伙伴们：

大家好！本周（M.DD-M.DD）主要围绕{核心工作线1}、{核心工作线2}等相关工作展开，{一句话概括本周重点}，现将具体情况总结如下：
 
一、核心任务
1. {工作线1名称}
    - 进展：
      - {子分类1}
        - {具体进展描述}；
        - {具体进展描述}；
      - {子分类2}
        - {具体进展描述}；
      - 待推进事项
        - {未完成的工作}；
    - 成果：
      - {成果1，概括性描述，突出价值}；
      - {成果2}；
    
2. {工作线2名称}
    - 进展：
      - {子分类1}
        - {具体进展描述}；
      - 待推进事项
        - {未完成的工作}；
    - 成果：
      - {成果描述}；
 
二、学习与思考
 
1. {主题}：{结合本周工作提炼的思考，150字左右}。
2. {主题}：{思考}。
3. {主题}：{思考}。 
 
三、下周工作计划
1. 【高优先级】{计划描述}。
2. 【高优先级】{计划描述}。
3. 【中优先级】{计划描述}。
4. 【低优先级】{计划描述}。


————————————

{config.author_name}

{config.company}

Mobile：{config.mobile}

E-mail：{config.email}

Ad：{config.address}

[图片]
```

### 写作风格要求

参照真实周报的语言风格，保持以下特征：

**开头段**：
- 固定格式"各位领导、管培生小伙伴们"
- 概括本周核心工作线（2-3个），加一句聚焦描述
- 例："聚焦任务落地与问题攻坚"、"聚焦核心任务落地，同步推进各项待办事项"

**核心任务-进展**：
- 按子分类组织，不要罗列日报原文
- 把同一项目不同天的碎片化进展合成连贯的段落描述
- 精炼概括：日报中的冗长描述要压缩提炼，去掉重复信息和过多细节，一句话能说清的不拆两句
- 用"完成XXX"、"推进XXX"、"明确XXX"等动词开头
- 技术细节保留但要概括，不要逐条复制日报
- "待推进事项"放在进展最后，用"未完成"、"暂未完成"等措辞

**核心任务-成果**：
- 高度概括，突出价值和意义
- 每条成果 1-2 句话
- 常用句式："{做了什么}，为{后续目标}奠定基础/提供支撑"

**学习与思考**：
- 3 条，每条一个小标题加冒号
- 结合本周具体工作提炼方法论
- 小标题用四字/六字总结如"追根溯源、精准突破"
- 每条 120-180 字，包含具体事例 + 总结感悟
- 语气正式但不空洞，要有真实工作的痕迹

**下周工作计划**：
- 从本周未完成项（❌）、进行中项（🔵）、待办字段智能推断
- 按优先级标注【高优先级】【中优先级】【低优先级】
- 高优先级：本周核心任务的延续/未完成项
- 中优先级：持续进行但非紧急的工作
- 低优先级：可延后或辅助性工作
- 每条用分号连接多个子任务

### 第 5 步：保存文件

将生成的周报保存到 skill 目录下的 `docs/` 子目录中，文件名格式：

```
{config.file_prefix}（M.DD-M.DD）.md
```

其中日期使用不带前导零的格式，如 `3.16-3.20`、`4.20-4.24`。

保存后告知用户文件路径，并询问是否需要调整。
